## Durable Memory Flush

- JT7 V1 has been explicitly refocused around intelligent Google Sheets tracker updates rather than broad autonomous job-search automation.
- Locked V1 operating model:
  - Gmail = intake layer for LinkedIn, Indeed, Otta, Greenhouse, Workday, recruiter emails
  - Calendar = interview/call truth and reinforcing evidence
  - Google Sheet = operational canonical tracker / system of record
  - manual logging = required for direct LinkedIn inbox conversations, phone calls, and off-platform decisions
- Locked V1 application status mapping:
  - alerts -> lead / later consolidated under Cold in the stricter schema discussion
  - confirmation -> Applied
  - recruiter outreach -> Recruiter update + Communication append, not automatic job stage unless explicit
  - interview scheduling -> Interviewing
  - rejection -> Rejected
  - offer -> Offer
- A major system consolidation pass selected a stricter shared schema and status model for the rebuilt JT7 core:
  - shared opportunity schema: company, role, status, last_activity, next_step, contact, source, thread_id, notes
  - allowed statuses from the consolidation pass: Applied, Recruiter Contacted, Screening, Interviewing, Offer, Rejected, Cold
- Layer ownership clarified during consolidation:
  - Intake Layer only collects and structures raw signals
  - Processing Layer owns interpretation, classification, matching, dedupe, next actions, and update-mode decisions
  - System of Record stores state only and must not interpret
  - Interaction Layer surfaces state/edits only and must not own processing logic
- Source-of-truth hierarchy clarified:
  - Google Sheets = operational source of truth for tracker state
  - Markdown docs = architecture, governance, memory, audits, and contracts
  - Gmail / Calendar / manual input = evidence inputs, not truth stores
- Current JT7 MVP tracking model was defined in five layers:
  - Foundation
  - Ingestion
  - Intelligence
  - Writeback
  - Operator Experience
- Current rough MVP snapshot discussed:
  - Foundation ~80%
  - Ingestion ~45%
  - Intelligence ~35%
  - Writeback ~35%
  - Operator Experience ~20%
  - Overall MVP ~40%
- Deployment should be tracked as a parallel workstream to MVP proof. Rough deployment readiness discussed at ~30%, with Telegram execution reliability as a major blocker.
- Current active milestone is still: prove one trustworthy end-to-end loop: Gmail search -> classify -> match -> tracker update.
