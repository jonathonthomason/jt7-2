# 2026-04-30

- Review Queue v1 was implemented and verified in `job-search-ui` with route/state/UI/test wiring; build and smoke test passed.
- JobOps was activated as a separate live Telegram bot/account on the same OpenClaw core, routed `jobops -> jobops`, with dedicated JobOps workspace files and operating-memory layer.
- Google Sheets remains canonical tracker truth; broad direct-board imports must go to staging rather than canonical `Jobs`.
- A new Staging Intake cockpit surface was added to `job-search-ui` so direct-board imports are visible separately from trusted Jobs.
- The staging surface was then improved with duplicate detection against canonical Jobs and stronger ranking/fit logic based on Jonathon’s search requirements.
- Platform should usually be the single checkpoint/resume surface; JobOps can update its own local docs, but Platform is the default final commit/push authority.
- Jonathon wants proactive warnings before quota-, rate-limit-, or state-mutating actions when a cheaper or durable-path answer may be sufficient.
- During the current budget-constrained build phase, JT7 should include budget-threshold warnings plus recommended wait time before further input when session/task exhaustion risk rises.
- A live JobOps validation loop on 2026-04-30 showed that JobOps can stay in-lane, preserve staging-vs-canonical discipline, and produce a mechanically stronger shortlist when prompted to return fit tier, duplicate risk, reasons for/against, and next operator action.
- A runtime-side staging writeback planner was added as the first safe bridge from local staging actions to canonical tracker behavior; it plans `create | merge | hold | reject` in dry-run form and must compare against trusted canonical jobs rather than local direct-import drift.
- `agents/jobops/ops/staging-queue-requirements.md` was promoted into a more actionable implementation artifact by mapping requirements across data model, ingestion logic, staging UI, and queue UI; the recommended build order is data model first, then ingestion, then staging UI, then queue UI.
